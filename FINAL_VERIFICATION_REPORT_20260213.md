| 条件 | 原报告值 | 体系逻辑计算值 | 差异 |
|------|---------|--------------|------|

### 根本原因分析

**为什么条件2和3的统计有严重偏差？**

1. **条件3差异最大(0.2% → 30.96%)**
   - SQL查询使用59个自然日，发现11个股票满足
   - 代码实际使用60个**交易日**，发现135个股票满足
   - 自然日vs交易日的时间跨度不同导致数据集不同

2. **条件2差异明显(0.5% → 4.36%)**
   - 成交量计算的MA20周期定义可能不同
   - 体系计算的vol_ratio是：当日成交量 / MA20成交量
   - 实际满足条件的股票远多于原报告

3. **条件1统计基本准确(0% → 0.69%)**
   - 只有3个股票满足（收盘价≥60日最高价）
   - 这是最严格的条件，满足率确实极低

### 关键发现：三个条件无交集

虽然有：
- ✓ 3个股票创出60日新高
- ✓ 19个股票成交量激增(≥1.5倍)
- ✓ 135个股票资金流为正

**但这三个集合完全没有交集**——没有任何股票同时满足全部三个条件！

因此，系统逻辑完全正确：
- 需要同时满足三个条件才能产生TRIGGER信号
- 由于无交集，所以正确地显示0个TRIGGER信号

根据 `pool_replay_engine/src/signals/trigger.py` 中的实现逻辑：

```python
is_breakout = (
    row.get("close", 0) >= row.get("high_60", 0)              # 条件1
    and row.get("vol_ratio", 0) >= cfg["vol_ratio_min"]        # 条件2
    and row.get("flow_3d", 0) > 0                              # 条件3
```

其中：
- **high_60**: 过去60个交易日(不含今天)的最高价
- **vol_ratio**: 当日成交量 / MA20成交量
- **flow_3d**: 过去3个交易日的资金流净额
